Memory Hierarchy and Optimization Reference

Quick reference for Module 8 topics: RAM/caches, locality, compiler optimizations, and measurement-driven tuning.

Hierarchy at a Glance

LevelTypical Property
RegistersSmallest, fastest
L1/L2/L3 cache (SRAM)Fast, limited capacity
Main memory (DRAM)Larger, slower
Disk/SSDPersistent, much slower

Locality Rules

  • Temporal locality: recently used data tends to be reused soon.
  • Spatial locality: nearby addresses tend to be accessed together.
  • Smaller stride generally improves spatial locality.
  • In C row-major arrays, varying the last index in inner loops often gives stride-1 access.

Cache Concepts

  • Hit: data found in cache; miss: fetch from lower level.
  • Common cache line size is 64 bytes.
  • Miss types: compulsory (cold), conflict, capacity.
  • Write policies: write-through vs write-back (dirty bit tracking).

Struct Layout and Padding

struct A {
    char  a;
    int   b;
    char  c;
    double d;
};
  • Compiler inserts padding to satisfy alignment constraints.
  • Ordering fields from larger to smaller often reduces padding.

Amdahl's Law

overall_speedup = 1 / ((1 - p) + p/s)
  • p: fraction of runtime affected.
  • s: speedup of optimized part.
  • If p is small, max whole-program speedup is limited.

GCC Optimization Levels

FlagUse
-O0No optimization; easiest debug mapping
-OgDebug-friendly optimizations
-O1Basic optimization
-O2Common production baseline
-O3More aggressive, may increase code size

Useful companions: -g for debug symbols, -pg for gprof instrumentation.

Optimization Workflow

  1. Get correct output first.
  2. Measure baseline.
  3. Profile hotspots.
  4. Optimize the hotspot.
  5. Re-measure and verify correctness.

Measuring Performance

$ /usr/bin/time -p ./app
real 1.23
user 1.18
sys  0.04
  • real: wall-clock elapsed time.
  • user: CPU time in user mode.
  • sys: CPU time in kernel mode.

gprof Quick Start

$ gcc -pg -O2 app.c -o app
$ ./app
$ gprof ./app gmon.out | less

Classic Loop Pitfalls and Fixes

// pitfall
for (int i = 0; i < strlen(s); i++) { ... }

// fix
size_t n = strlen(s);
for (size_t i = 0; i < n; i++) { ... }

Cache-Friendly Matrix Tips

  • Prefer loop orders with stride-1 access in innermost loops.
  • Use blocking/tiling so working submatrices fit cache.
  • Avoid unnecessary temporary arrays and scattered access patterns.

Pointer Aliasing Reminder

Compiler must assume pointers may alias unless it can prove otherwise. This can block optimizations that would otherwise be legal.

Common Mistakes Checklist

  • Optimizing before profiling.
  • Comparing only single-run timings.
  • Ignoring cache behavior and memory access patterns.
  • Changing algorithms for tiny gains that hurt readability without measured value.
  • Assuming source-level simplification always maps to faster machine code.
