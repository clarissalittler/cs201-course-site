Concurrency and Threads Reference

Quick reference for Module 6 topics: pthreads, race conditions, mutexes, condition variables, semaphores, and deadlock patterns.

Compile/Link Flags

$ gcc -pthread -O2 -g program.c -o program

Use -pthread for correct compile + link behavior with pthreads.

Create and Join Threads

void *worker(void *arg) {
    return NULL;
}

pthread_t tid;
pthread_create(&tid, NULL, worker, arg_ptr);
pthread_join(tid, NULL);
  • Start routine signature: void *fn(void *arg).
  • Passing pointer to loop variable is a common race bug; pass stable per-thread data.

Race Condition Pattern

// not atomic:
counter++;
// often compiles to load, increment, store

Two threads interleaving these steps can lose updates.

Mutex Basics

pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;

pthread_mutex_lock(&m);
// critical section
pthread_mutex_unlock(&m);
  • Always unlock on all return/error paths.
  • pthread_mutex_trylock() tests lock availability without blocking.

Condition Variables

pthread_mutex_lock(&m);
while (!condition_is_true) {
    pthread_cond_wait(&cv, &m);
}
// condition holds
pthread_mutex_unlock(&m);
  • Use while, not if, because of spurious wakeups.
  • pthread_cond_wait atomically releases the mutex and sleeps, then re-locks on wake.
  • Use pthread_cond_signal (wake one) or pthread_cond_broadcast (wake all).

Semaphores

sem_t s;
sem_init(&s, 0, initial_value);
sem_wait(&s);   // P operation (decrement/block)
sem_post(&s);   // V operation (increment/wake)
sem_destroy(&s);
  • Binary semaphore (initial_value=1) can enforce mutual exclusion.
  • Counting semaphore tracks available units/resources.

Producer/Consumer Semaphore Pattern

sem_t empty, full, mutex;
// empty = N, full = 0, mutex = 1

// producer
sem_wait(&empty);
sem_wait(&mutex);
// insert item
sem_post(&mutex);
sem_post(&full);

// consumer
sem_wait(&full);
sem_wait(&mutex);
// remove item
sem_post(&mutex);
sem_post(&empty);

Deadlock: Four Conditions

  • Mutual exclusion
  • Hold and wait
  • No preemption
  • Circular wait

Most practical prevention in systems code: enforce a global lock acquisition order.

Reader-Writer Lock

pthread_rwlock_t rw = PTHREAD_RWLOCK_INITIALIZER;

pthread_rwlock_rdlock(&rw);  // many readers can hold concurrently
pthread_rwlock_unlock(&rw);

pthread_rwlock_wrlock(&rw);  // writer is exclusive
pthread_rwlock_unlock(&rw);

Common Mistakes Checklist

  • Sharing stack addresses that go out of scope with worker threads.
  • Returning with a mutex still locked.
  • Using if around pthread_cond_wait instead of while.
  • Assuming operation order without synchronization.
  • Inconsistent lock ordering across code paths (deadlock risk).
  • Using volatile as a replacement for mutex/atomic synchronization.
