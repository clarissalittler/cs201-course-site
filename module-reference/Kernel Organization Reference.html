Kernel Organization Reference

Quick reference for Module 10 topics: kernel responsibilities, privilege transitions, architecture styles, and practical design tradeoffs.

What the Kernel Does

  • Manages CPU scheduling and context switches.
  • Manages virtual memory, page tables, and protection.
  • Provides file systems, networking, and device access.
  • Exposes services via system calls.
  • Enforces isolation between processes and between user/kernel mode.

Privilege Levels

ModeTypical Capabilities
User mode (ring 3)No direct privileged instructions or device control
Kernel mode (ring 0)Full privileged access to CPU/MMU/devices

System Call Path (x86-64 Linux)

  1. User code places syscall number in %rax and args in syscall registers.
  2. CPU executes syscall and switches to kernel mode.
  3. Kernel dispatches through syscall table to handler.
  4. Kernel returns result in %rax and resumes user mode.

Common user-space wrapper path: libc wrapper -> syscall instruction -> kernel handler -> return.

Context Switch Essentials

  • Save current thread/process register state.
  • Select next runnable task.
  • Restore next state and memory mapping context.
  • Switching tasks incurs overhead (cache/TLB disruption + scheduler work).

Kernel Architecture Styles

StyleCore IdeaStrengthTradeoff
MonolithicMost services run in kernel spaceHigh performance via direct callsLarge trusted code base; faults can be catastrophic
MicrokernelMinimal kernel, services in user spaceFault isolation and smaller trusted coreMore IPC/mode-switch overhead
HybridMix of both approachesPragmatic performance/reliability balanceArchitectural complexity

Monolithic Kernel Notes

  • Linux is monolithic with loadable kernel modules.
  • Drivers, file systems, scheduler, networking, and VM all run in privileged space.
  • Module loading improves deployability, not protection isolation.

Microkernel Notes

  • Kernel keeps only essentials: low-level scheduling, memory primitives, IPC.
  • Drivers and services run as user-space servers.
  • If a server crashes, system can often recover without full kernel panic.

IPC and Message Passing

  • Microkernels rely heavily on IPC for service composition.
  • Fast IPC design is central to microkernel performance.
  • Copying and context switches can dominate if IPC path is inefficient.

Reliability and Security Design Points

  • Smaller trusted computing base can simplify auditing and formal verification.
  • Compartmentalization limits blast radius of component bugs.
  • Monolithic kernels reduce boundary crossings but increase privileged attack surface.

Modern Practical Directions

  • Containers: process isolation using kernel namespaces/cgroups.
  • eBPF: verified bytecode executed in-kernel for observability/networking.
  • Rust in kernel components: memory-safety improvements for new code.
  • Specialized kernels (e.g., RTOS/microkernels) in safety-critical systems.

Common Mistakes Checklist

  • Assuming loadable modules are isolated like user-space processes.
  • Treating all context switch costs as constant; cache/TLB effects matter.
  • Assuming microkernels are always too slow for real systems.
  • Assuming monolithic kernels are inherently insecure; engineering quality still dominates outcomes.
  • Ignoring workload: architecture choice depends on goals (latency, throughput, isolation, certifiability).
