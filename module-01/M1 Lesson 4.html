Lesson 4: Byte Ordering

Numeric Representation: Byte Ordering

Byte order is the order in which a multi-byte data type is stored in memory. These are stored as a consecutive set of bytes that are laid out from some low address to a high address, with the lowest address representing the "address" of this data.

There are two ways to lay out the bit pattern of a value across several bytes. One way is to make the most significant byte (the leftmost byte of the bit pattern) be stored starting at the high address. This is called little endian, and I won't get into why it's called this (the textbook talks about this weird name). The other way is to store the most significant 8 bits at the low address. This is called big endian.

For an example of byte ordering, if we are storing an integer, we need 4 consecutive bytes in memory. Note that for brevity, we will be showing each byte as a two-digit hex number. For example, if we have the hex number 0x12345678 to store, the bytes will be stored in 'big endian' as:

  address ->  200  201  202  203
              12    34   56   78

For little endian, the bytes will be stored:

  address ->  200  201   202 203
               78   56   34   12

See the diagrams below for an illustration of the above.

Aside: Most computers you will work with are (or are a copy of) Intel architecture which is little endian. Some high-end commercial systems by IBM and Oracle may still be big-endian by default (or bi-endian where they can switch between the two), but all consumer electronics today are 'little endian' architecture because of the popularity of the Intel architecture.

On the other hand, bit transmissions on the Internet are basically big-endian. So often, bits being sent between network cards need to be re-arranged when they are eventually stored in the host's memory.

For more information on the history of byte ordering, you can optionally read On Holy Wars and a Plea for Peace [link opens in a new tab].

The byte order usually doesn't affect programmers unless you are working across multiple bytes at the bit pattern level. This is pretty rare; you might run into this issue if you are doing very low-level or machine code level programming.

Byte ordering can be an issue when communicating data from one machine to another over a network connection. For more information about this, check the Wikipedia entry for Endianness [link opens in new tab].

Aside: If you print out a little-endian machine's byte-ordered value in the order of increasing addresses (which is how a char array is stored), each byte will be written in reverse order. If you want to see the actual bit pattern of a number in the right order on a 'little endian' machine, you can change 'showbytes' function to print out the char bytes from len-1 to 0:

for (i = len-1; i > 0; i--) 
 

This will save you some time in deciphering the number in that you don't have to first reverse the hex digits to get the actual bit pattern from the print out.

This showbytes.c code file has this function in it. 

#include <stdio.h>
#include <string.h>

typedef unsigned char * byte_pointer;

void show_bytes(byte_pointer start, size_t len)
{
   int i;
   for(i=0; i<len; i++)
       printf(" %.2x", start[i]);
   printf("\n");
}



void show_int(int x)
{
    show_bytes((byte_pointer) &x, sizeof(int));
}

void show_float(float x)
{
    show_bytes((byte_pointer) &x, sizeof(float));
}

void show_pointer(void *x)
{
   show_bytes((byte_pointer) &x, sizeof(void *));
}

void test_show_bytes(int val)
{
   int ival = val;
   float fval = (float) val;
   int *pval = &ival;
   
   show_int(ival);
   show_float(fval);
   show_pointer(pval);
  
}


int main()
{
    const char *sval = "12345";
   //int i =-23432;

   //test_show_bytes(i);   
    show_bytes((byte_pointer) sval, strlen(sval)+1);
    return 0;
}
