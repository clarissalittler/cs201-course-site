Module 8 Summary

Wrap Up

In this module, we studied memory from the hardware level up through the programmer's interface. We began with the underlying storage technologies — SRAM and DRAM — and saw how the enormous speed gap between the CPU and main memory motivates the entire memory hierarchy. We then explored how the principle of locality allows caching to be effective, and how cache organization (direct-mapped, set-associative) determines hit rates and performance. From there, we moved to virtual memory, the operating system's abstraction that gives each process a private address space, isolates processes from each other, and uses disk as overflow when physical memory is full. Finally, we studied dynamic memory allocation: how malloc and free manage the heap, the data structures and algorithms they use internally, and the common programming errors that arise from manual memory management in C.

Key Takeaways

  • The memory hierarchy is a series of storage levels (registers, L1/L2/L3 caches, main memory, disk) where each level serves as a cache for the level below. Faster levels are smaller and more expensive per byte; slower levels are larger and cheaper. Locality is what makes this hierarchy effective.
  • Locality comes in two forms: temporal (recently accessed data will likely be accessed again soon) and spatial (data near recently accessed data will likely be accessed soon). Writing code with good locality is one of the most impactful optimizations a programmer can make.
  • Caches exploit locality by storing recently used data in fast SRAM close to the CPU. Cache organization — including cache lines, sets, mapping schemes, and write policies — determines how effectively the cache can reduce average memory access time.
  • Virtual memory is an abstraction provided by the OS and hardware together. Each process sees a private, contiguous virtual address space. The MMU translates virtual addresses to physical addresses using page tables. The TLB caches recent translations to make this fast. Page faults handle the case when a virtual page is not in physical memory, loading it from disk on demand.
  • Dynamic memory allocation with malloc/free gives the programmer explicit control over heap memory. Internally, malloc maintains a free list of available blocks, uses allocation strategies (first fit, best fit, next fit) to choose blocks, and coalesces adjacent free blocks to fight fragmentation.
  • Common memory errors in C — memory leaks, dangling pointers, double frees, and buffer overflows — are a major source of bugs and security vulnerabilities. Tools like valgrind can detect these errors and should be used routinely on any C program that uses dynamic memory allocation.
