Signals in Linux Systems

Signals are higher-level or software-level exceptions that processes, including the operating system, can send to other processes (or to itself). These can be 'caught' by the process through a signal handler: a function that is to be called when the signal is caught by the process. If the process doesn't have a signal handler, the operating system has a default action that it takes.

The list of POSIX signals on the Signals (IPC) wiki shows all of the 30 signal types that can be sent in a Linux system. You don't need to memorize these signal types. Below is a list of some important signals:

  • SIGINT (2): default action: terminates
    this is sent to the foreground process running in a terminal when the user hits <ctrl> c. Also, can be sent to any user process by using the command "kill -2".
  • SIGFPE (8): default action: terminates
    Called a 'floating point exception', it is sent when dividing by 0 is attempted. Ironically, by default, floating point divide by zero isn't affected by this signal, only integers are.
  • SIGKILL (9): default action: terminates
    can send this by using the command kill -9 followed by the PID of the process. This is one of two signals that can't be overridden by a signal handler.
  • SIGSEGV (11): default action: terminates
    the infamous 'segmentation fault' when accessing invalid memory.
  • SIGALRM (14): default action: terminates
    can be used to set up a timer. A timer sends this signal repeatedly at every n intervals of time.
  • SIGCHLD (17): default action: ignores
    sent from a child process to its parent when the child terminates or is stopped.
  • SIGCONT (18): default action: ignores
    Sent to continue running a stopped process.
  • SIGSTOP (19): default action: stops until SIGCONT
    sent to a process to stop or suspend a process. This is NOT sent from the terminal when hitting <ctrl> z. it's typically used by the operating system, and its behavior cannot be changed by a signal handler.
  • SIGTSTP (20): default action: stops until SIGCONT
    Same as SIGSTOP except sent from a terminal. This IS sent from the terminal when hitting <ctrl> z.

We'll discuss how a process can deal with some of these in a later lecture.
