Module 9 Summary

Wrap Up

In this module, we studied how the operating system manages memory and how programs are loaded, linked, and given access to the heap. We began with virtual memory — the abstraction that gives each process a private address space while sharing physical memory. We then examined linking: how the linker resolves symbols, combines object files, and supports shared libraries through position-independent code. From there, we studied dynamic memory allocation with malloc and free, including the internal data structures that manage the heap. We concluded with garbage collection, the automatic alternative to manual memory management used by most modern languages.

Key Takeaways

  • Virtual memory is an abstraction provided by the OS and hardware together. Each process sees a private, contiguous virtual address space. The MMU translates virtual addresses to physical addresses using page tables. The TLB caches recent translations to make this fast. Page faults handle the case when a virtual page is not in physical memory, loading it from disk on demand.
  • Linking is the process of combining object files into an executable. The static linker resolves symbol references, handles strong vs. weak symbols, and produces a self-contained binary. The dynamic linker defers library loading until runtime, using the PLT and GOT to resolve function addresses lazily. Dynamic linking saves disk space and memory when multiple programs share the same libraries.
  • Position-independent code (PIC) allows shared libraries to be loaded at any virtual address. This is essential for ASLR and for sharing physical pages of library code between processes.
  • Dynamic memory allocation with malloc/free gives the programmer explicit control over heap memory. Internally, malloc maintains a free list of available blocks, uses allocation strategies (first fit, best fit, next fit) to choose blocks, and coalesces adjacent free blocks to fight fragmentation.
  • Common memory errors in C — memory leaks, dangling pointers, double frees, and buffer overflows — are a major source of bugs and security vulnerabilities. Tools like valgrind can detect these errors and should be used routinely on any C program that uses dynamic memory allocation.
  • Valgrind is a dynamic analysis tool that detects memory errors at the exact instruction that causes them. Running programs under Valgrind with --leak-check=full catches leaks, invalid reads and writes, use of uninitialized memory, and double frees. A clean Valgrind report should be the standard for any C program that uses dynamic memory.
  • Garbage collection automates memory management. Mark-and-sweep traces reachable objects from the roots and frees everything else. Reference counting is simpler but cannot handle cycles. Generational collectors exploit the observation that most objects die young, collecting the nursery frequently and the old generation rarely.
