How to Measure Performance

In order to understand how well our optimization is working, we need to be able to measure how fast our programs are running. The textbook defines something called cycles per element (CPE) which it then uses for its optimization example. This is fine but not necessarily available outside of the book's code.

Most optimization tools in the industry (benchmarks, profilers, etc) use CPU time. CPU time is the amount of time in seconds a CPU was processing this particular program (as opposed to other programs that were also using the CPU). This is more accurate than straight clock time as it doesn't measure the times that the program was inactive.

In your lab this week, you can use the command time at the terminal to get CPU time. For example, if we want to know how much CPU time it takes to run a program called report, we type at the command line:

time ./report 
 

Note that we need to use ./ to indicate where the executable is. The output will be whatever the output the program produces followed by three lines:

....
    real    0m6.480s
    user    0m6.448s
    sys     0m0.032s 


The first value is overall CPU time. The other values are how much of the CPU time was spent in user mode vs. kernel (system) mode. The command is very useful for getting an overall feel for how fast your program is running but has it's downside in that it doesn't say where in the code any slowdowns are. this is where using a profiler would be useful.
