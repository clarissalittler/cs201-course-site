Lesson 2: Exceptions

Exceptions

The events that cause ECF to happen are called generically exceptions. An exception is an abrupt change in the control flow in response to some change in the processor’s state. The event might be directly related to the execution of the current instruction, such as a segmentation fault because the software was accessing a NULL pointer. On the other hand, the event might be unrelated to the execution of the currently running program, such as a signal that data from the disk is ready to be read by another program.

How Exceptions are Handled

When an exception happens, the hardware looks up an exception handler. This is a piece of code for managing a particular exception. Exception handlers are usually located in the operating system's code.

The start instruction for this code is stored in an exception table. The exception table is indexed by an exception number. When an exception happens, the hardware is given this number which it uses to look up which handler to run in the exception table. Exception numbers are assigned by either the operating system or by the hardware.

The hardware then transfers control flow to this handler. The handler is similar to a procedure call except the return address isn't necessarily the next instruction. Some additional processor state is also pushed onto either the program's stack or the kernel's stack.

When the handler is done, three things can happen depending on the exception. It can continue to run the program either starting at the same instruction or at the next instruction. It can also terminate the program (aborting) giving back control to the operating system.
