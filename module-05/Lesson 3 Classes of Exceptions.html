Lesson 3: Classes of Exceptions

Classes of Exceptions

There are four classes of exceptions:

  • Interrupts: Interrupts occur asynchronously (basically at any time) as a result of signals from I/O devices that are external to the processor. Hardware interrupts are asynchronous in the sense that they are not caused by the execution of any particular instruction. These can happen in the middle of processing the current instruction in the CPU. The hardware signals the CPU by setting a 'pin' on the microchip that the CPU will read when it is done processing the current instruction. The hardware also sends the exception number on the system bus to let the CPU know which exception to run. Exception handlers for hardware interrupts are often called interrupt handlers.
  • Traps: This is one of three other synchronous exceptions that happen as a result of the current instruction (sometimes called the faulting instruction). Traps however are planned intentional exceptions not necessarily caused by an error.
    A system call (a call to a function that asks for a service to be performed by the operating system) is one common type of trap. System calls can be asking to access the disk, such as in a read() function, loading a new program or asking to prematurely terminate a program. In assembly code, there is the special instruction syscall that allows a program to ask for services from the operating system (the syscall number is placed in %rax before executing the instruction). This causes a trap to an exception handler that decodes the given arguments and runs the appropriate kernel routine.
  • Faults: Faults result from error conditions. They are also synchronous but might not be intentional on the part of the programmer. Control is passed to a fault handler that then tries to correct the problem. If it can do so, then it passes control to the same instruction so it can re-run. Otherwise it will stop the process (do an abort).
  • Aborts: These are also unintentional errors caused by the current instruction (so they are synchronous). But unlike faults, there is no way to recover from them and so the program is terminated. Usually these are hardware-related failures, such as a machine check exception caused by corrupted memory or a failed hardware component. (Note: integer divide by zero is technically classified as a fault, not an abort, though on Linux it typically terminates the process via SIGFPE.)
